# 에이전트 테스트 문구

아래는 지금 구성한 `dbt-orchestrator + dbt-discoverer` 구조를 OpenCode에서 실제로 테스트할 때 사용할 수 있는 추천 테스트 문구들입니다.

핵심은 다음을 검증하는 것입니다.

```text id="jlwmwf"
1. dbt-orchestrator가 primary agent로 동작하는가?
2. dbt-discoverer를 적절한 시점에 호출하는가?
3. discoverer 결과를 orchestration에 활용하는가?
4. 최종 응답이 단순 중계가 아니라 상태 기반 판단인가?
5. source → staging → mart 흐름을 이해하는가?
6. output contract가 유지되는가?
```

---

# 1. 가장 기본적인 orchestration 테스트

## 테스트 문구

```text id="jlwmwg"
orders 관련 staging 모델을 만들어줘.
현재 PostgreSQL raw schema 기준으로 판단해줘.
```

---

## 기대 동작

`dbt-orchestrator`가:

```text id="jlwmwh"
1. "현재 PostgreSQL raw schema 기준"
→ dbt-discoverer 호출 필요 판단

2. discoverer 결과 확인
- raw.orders 존재 여부
- source.yml 등록 여부
- stg_orders 존재 여부

3. 필요 시 source 설계 제안

4. stg_orders.sql 생성 방향 제시
```

---

# 2. discoverer 자동 호출 테스트

## 테스트 문구

```text id="jlwmwi"
현재 DB 상태 기준으로 누락된 staging 모델이 있는지 확인해줘.
```

---

## 기대 동작

`dbt-orchestrator`가:

```text id="jlwmwj"
1. dbt-discoverer 호출
2. raw table ↔ staging model coverage 비교
3. 누락 모델 식별
4. 다음 작업 우선순위 제시
```

---

# 3. orchestration 품질 테스트

## 테스트 문구

```text id="jlwmwk"
mart_customer_orders 모델을 만들고 싶어.
현재 프로젝트 상태 기준으로 필요한 선행 작업도 같이 알려줘.
```

---

## 좋은 응답 특징

좋은 orchestrator는 단순 SQL 생성이 아니라:

```text id="jlwmwl"
- staging coverage 확인
- 필요한 source 확인
- grain 제안
- join key 확인
- 누락된 모델 식별
- 테스트 필요성 제안
```

까지 해야 합니다.

---

# 4. dbt-discoverer 결과 활용 테스트

## 테스트 문구

```text id="jlwmwm"
현재 raw.orders와 raw.order_items 구조를 기준으로
어떤 mart 모델을 만들 수 있는지 추천해줘.
```

---

## 기대 포인트

여기서 중요한 건:

```text id="jlwmwn"
discoverer 결과를
"상태 정보"
→ "모델링 전략"
으로 변환하는가?
```

입니다.

---

# 5. 상태 기반 orchestration 테스트

## 테스트 문구

```text id="jlwmwo"
현재 dbt 프로젝트에서
가장 우선적으로 해야 할 작업 3가지를 추천해줘.
```

---

## 기대 동작

좋은 orchestrator는:

```text id="jlwmwp"
1. discoverer 호출
2. coverage 분석
3. missing source 식별
4. missing staging 식별
5. missing tests 식별
6. 우선순위 결정
```

을 해야 합니다.

---

# 6. dbt-debug-helper 연동 테스트

## 테스트 문구

```text id="jlwmwq"
dbt run 실행 시
relation "stg_order" does not exist 오류가 발생했어.
원인 분석해줘.
```

---

## 기대 동작

좋은 orchestrator는:

```text id="jlwmwr"
1. ref('stg_order') 오타 가능성 확인
2. 실제 모델 존재 여부 확인
3. discoverer로 현재 모델 상태 재탐색
4. debug-helper 호출
5. 수정 방향 제안
```

을 해야 합니다.

# 7. output contract 검증 테스트

## 테스트 문구

```text id="jlwmws"
현재 PostgreSQL raw schema와
dbt 프로젝트 상태를 다시 탐색해줘.
```

## 기대 결과

```text id="jlwmwt"

3. dbt Mapping Status

5. Recommended Next Actions
```

형식을 유지하는지 확인합니다.

```text id="jlwmwu"

```

입니다.

# 8. 컨텍스트 부족 테스트

## 테스트 문구

```text id="jlwmwv"
orders 모델 만들어줘.
```

---

## 기대 포인트

좋은 orchestrator는 바로 SQL 생성하지 않고:

```text id="jlwmww"

- existing staging인지?
- mart인지?
- source 등록 여부는?
```

```text id="jlwmwx"

```

을 테스트하는 겁니다.

# 10. 가장 중요한 테스트

이 테스트가 핵심입니다.

## 테스트 문구

```text id="jlwmx0"
현재 상태 기준으로
orders 관련 mart 모델을 안정적으로 만들기 위해
먼저 확인해야 하는 사항들을 정리해줘.
```

---

## 왜 중요하냐면

이 테스트는:

```text id="jlwmx1"
- orchestration
- state awareness
- dependency understanding
- context refresh
- subagent integration
```

모두를 동시에 검증합니다.

---

# 좋은 orchestrator 응답 예시

좋은 응답은 보통 이런 흐름입니다.

```text id="jlwmx2"
1. 현재 상태를 먼저 확인해야 합니다.
2. dbt-discoverer 기준으로 다음을 확인합니다.
   - raw.orders 존재 여부
   - raw.order_items 존재 여부
   - source 등록 상태
   - staging coverage
3. 현재 stg_order_items가 누락되어 있다면
   mart 생성 전에 먼저 staging 모델 생성이 필요합니다.
4. 이후 grain(customer/day/order 기준)을 정의해야 합니다.
5. 그 다음 mart 모델 설계를 진행하는 것이 안전합니다.
```

```text id="jlwmx3"

subagent 결과를
현재 상태 모델로 해석했는가?
⭕ 핵심
```

즉 테스트의 본질은:

```text id="jlwmx4"
"Primary agent가 상태 기반 판단을 하는가?"
```

입니다.
