# 프로젝트 킥오프 R&R 및 액션아이템

## 1. 목적

프로젝트 시작 단계에서 팀원 간 역할과 책임(R&R)을 먼저 고정하고, 초기 실행 액션 아이템을 한 문서에서 관리한다.

## 2. 적용 범위

- DBT 전환 프로젝트 킥오프
- 분석, 설계, 개발, 테스트, 이관 전 구간의 내부 실행 관리
- 고객 승인 문서가 아닌 팀 내부 운영 문서

## 3. 팀 운영 원칙

| 원칙 | 설명 |
|---|---|
| 역할 고정 | 누가 결정하고, 누가 구현하고, 누가 검증하는지 초기에 명확히 한다. |
| 액션 단위 관리 | 회의 결과는 반드시 담당자와 기한이 있는 액션 아이템으로 변환한다. |
| 병목 분리 | 승인자와 실무자를 가능한 분리한다. |
| 이슈 가시화 | 보류 항목과 리스크는 별도 이슈로 남긴다. |

## 4. R&R 정의

| 인력 | 주요 R&R | 세부 책임 |
|---|---|---|
| 김무경 | 최종 의사결정/승인 | 범위 확정, 기준 승인, 우선순위 조정 |
| 박효진 | PMO/리스크/운영조율 | 일정 조율, 이슈 추적, 회의체 운영 |
| 김지훈 | 구현 리드 | 핵심 모델 전환, 복합 로직 처리 |
| 김재웅 | 구현 보조/예외 처리 | 단순 JOB, 수정 반영, 재처리 대응 |
| 김진형 | 기술 검증/구조 품질 | lineage, 정합성, 구조 리뷰, 성능 관점 검토 |
| 최경연 | 산출물/문서/로그 정리 | 결과서, 템플릿, 교육자료, 로그 취합 |

## 5. 커뮤니케이션 룰

| 항목 | 기준 |
|---|---|
| 정기 회의 | 주 1회 이상 |
| 회의 결과 | 결정 사항, 보류 사항, 담당자, 기한을 남긴다. |
| 이슈 관리 | 구두 합의는 원칙적으로 이슈로 재기록한다. |
| 변경 관리 | 범위/기준 변경은 승인자 확인 후 반영한다. |

## 6. 초기 액션 아이템

| No | 액션 아이템 | 담당 | 기한 | 상태 | 비고 |
|---|---|---|---|---|---|
| 1 | 프로젝트 목표 및 성공 기준 확정 | 김무경, 박효진 | 착수 1주차 | 진행 전 | 킥오프 합의 필요 |
| 2 | 단계별 R&R 최종 확인 | 전원 | 착수 1주차 | 진행 전 | 역할 충돌 점검 |
| 3 | 작업용 산출물 정의표 작성 | 최경연, 박효진 | 착수 1주차 | 진행 전 | 문서 템플릿 포함 |
| 4 | 공식 산출물-작업용 산출물 매핑표 작성 | 최경연, 김진형 | 착수 1주차 | 진행 전 | 문서 단위 매핑 |
| 5 | WBS 산출물명 정리 | 김진형, 김지훈 | 착수 2주차 | 진행 전 | 공식 산출물명 기준 |

## 7. 초기 체크포인트

| 시점 | 확인 항목 |
|---|---|
| 1주차 | 역할 배분, 회의체, 산출물 체계 확정 |
| 2주차 | 주요 작업용 산출물 초안 생성 |
| 3주차 | 공식 산출물 승격 기준 점검 |

## 8. 이슈 에스컬레이션 기준

- 24시간 이상 담당자 판단이 어려운 항목
- 범위 변경이 수반되는 항목
- 고객 승인 전제가 필요한 항목
- 일정 영향이 1주 이상 예상되는 항목
