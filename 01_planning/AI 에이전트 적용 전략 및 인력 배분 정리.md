# AI 에이전트 적용 전략 및 인력 배분 정리

## 1. 논의 배경

DBT 전환 WBS는 일반적인 수작업 기준으로 보면 공격적인 일정이지만, AI 에이전트를 변환과 검증에 적극 활용한다는 전제를 두면 현실성이 높아진다.

## 2. 핵심 전제

| 항목 | 내용 |
|---|---|
| AI 적용 범위 | 변환 + 검증 |
| 검증 범위 | `row count`, `key integrity`, `column sample`, `aggregation consistency` |
| 오류 로그 | 구조화된 JSON |
| AI 수정 범위 | SQL, dbt model, macro |
| 검증 규칙 수정 | 사람 피드백 후 반영 |
| 최종 책임 | 사람 승인 |

## 3. AI 에이전트 적용 구조

| 단계 | 수행 주체 | 내용 |
|---|---|---|
| 1 | 사람 | 모델별 검증 기준치 사전 정의 |
| 2 | AI | 검증 수행 |
| 3 | AI | 검증 결과 JSON 로깅 |
| 4 | AI | 오류 로그 기반 모델 수정 |
| 5 | AI | 수정 후 재검증 |
| 6 | 사람 | 기준 충족 여부 최종 승인 |

## 4. 일정 현실성 판단

| 구분 | 판단 |
|---|---|
| 개발/구현 | AI 활용으로 일정 압축 가능 |
| 검증/통합 테스트 | AI를 써도 사람 승인 병목 존재 |
| 전체 일정 성패 | AI 성능보다 기준/로그/승인 체계에 좌우 |
| 종합 판단 | 공격적이지만 논리적으로 성립 가능 |

## 9. 병목 최소화 원칙

| 원칙 | 설명 |
|---|---|
| 역할 분리 | 결정, 구현, 검증, 문서를 분리 |
| 승인자 분리 | 승인자는 실무 구현에서 제외 |
| 검증 분리 | 검증 기준 정의와 검증 실행을 분리 |
| 문서 분리 | 문서 작업이 개발자 시간을 잡아먹지 않도록 분리 |

## 10. 결론

이 일정은 AI를 단순 보조가 아니라 실행과 검증까지 포함한 핵심 전제로 볼 때 충분히 논리적이다.

다만 성공 조건은 다음 3가지다.

| 성공 조건 | 설명 |
|---|---|
| 검증 기준의 명확성 | 모델별 판단 기준이 분명해야 함 |
| JSON 로그 표준화 | AI 재수정 품질을 좌우함 |
| 사람 승인 절차 | 예외와 최종 책임을 통제함 |
