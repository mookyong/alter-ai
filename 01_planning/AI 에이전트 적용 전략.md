# AI 에이전트 적용 전략

## 1. 목적

DBT 전환 작업에서 AI 에이전트를 단순 보조 도구가 아니라, 변환과 검증을 함께 수행하는 실행 주체로 활용한다.

핵심 목표는 다음과 같다.

- 단순 변환 작업 자동화
- 모델별 검증 자동화
- 오류 로그 기반 재수정 루프 구성
- 사람은 기준 정의와 최종 승인에 집중

## 2. 적용 전제

본 전략은 다음 전제를 기반으로 한다.

- 단순 SQL, 공통 테이블, 반복 패턴 JOB은 AI가 높은 비율로 자동 변환 가능
- 검증도 AI가 적극 수행 가능
- 검증 기준은 사전에 정의되어 있어야 함
- 최종 책임은 사람이 보유

## 3. 검증 범위

검증 기준은 다음 항목을 모두 포함한다.

- `row count`
- `key integrity`
- `column sample`
- `aggregation consistency`

즉, 단순 건수 검증뿐 아니라 키 무결성, 컬럼 샘플, 집계 일관성까지 포함하는 통합 검증 체계로 운영한다.

## 4. 실행 구조

AI 에이전트는 아래 순서로 동작한다.

1. 모델별 검증 기준치 참조
2. 검증 수행
3. 결과를 구조화된 JSON으로 기록
4. 오류 발생 시 해당 로그를 기반으로 모델 수정
5. 수정 후 재검증 수행
6. 기준 충족 시 사람 승인 단계로 이관

## 5. 오류 로그 원칙

오류 로그는 가능한 한 구조화된 JSON 형식으로 남긴다.

이유는 다음과 같다.

- AI가 재수정에 활용하기 좋음
- 오류 유형별 분류가 용이함
- 반복 오류 패턴 추적이 가능함
- 후속 분석과 리포팅 자동화가 쉬움

## 6. 수정 범위

AI의 수정 범위는 전체로 본다.

- SQL
- dbt model
- macro

다만 검증 규칙 자체의 자동 수정은 하지 않는다.

검증 규칙은 사람의 피드백 이후에만 수정한다.

## 7. 일정에 대한 해석

이 전략을 적용하면, WBS 일정은 단순한 개발 일정이 아니라 **AI 기반 자동화 루프를 전제로 한 일정**으로 해석해야 한다.

따라서 일정의 현실성은 다음에 달려 있다.

- 검증 기준치의 명확성
- JSON 로그 표준화 수준
- 재수정 루프의 안정성
- 예외 케이스에 대한 사람 승인 체계

## 8. 주요 리스크

- 검증 기준이 느슨하면 자동화 효과가 감소함
- JSON 스키마가 흔들리면 AI 재수정 품질이 저하될 수 있음
- 예외 케이스가 많아지면 사람 개입 비중이 다시 증가함
- 검증 규칙 변경이 잦으면 운영 병목이 생길 수 있음

## 9. 결론

본 일정은 AI 활용을 전제로 할 때 충분히 논리적인 계획이다.

특히 변환과 검증을 함께 자동화하는 구조라면, 기존의 수작업 중심 전환보다 일정 압축 여지가 크다.

다만 성공 조건은 AI 자체보다 **검증 기준의 명확성, 로그 구조화, 사람 승인 절차**에 있다.
